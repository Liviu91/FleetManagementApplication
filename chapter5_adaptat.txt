Capitolul 5.  Proiectare de detaliu și implementare
5.1. Arhitectură generală și prezentare
Aplicația în cauză e o aplicație care poate fi extinsă la nivel enterprise și folosită, deci, de un număr foarte mare de utilizatori. Întregul sistem este gândit de așa natură încât să vină la pachet cu performanța necesară și capacitatea de extindere necesară pentru a deservi o întreagă flotă de mașini.
În acest scop, întregul sistem se împarte în 3 mari componente: 
    • Componenta de Web
    • Componenta pentru Mobile
    • Componenta de Worker
Componenta de Mobile este reprezentată de aplicația Android dezvoltată cu ajutorul framework-ului .NET MAUI și are o funcționalitate care se ocupă de partea de colectare de date de la mașină și GPS-ul telefonului plus partea de publicare de mesaje în cozile broker-ului RabbitMQ. 
Componenta de Worker reprezintă o aplicație separată care interacționează cu RabbitMQ, preluând mesajele din cozi, procesându-le și injectându-le în baza de date din serverul SQL.
Componenta de Web reprezintă ultima parte implicată în întregul proces și realizează procesarea datelor din baza de date și afișarea acestora pe interfața UI. Acest proces este detaliat în subcapitolul următor însă menționăm acum, ca elemente principale din cadrul acestei componente:
    • aplicația de MVC(Model-View-Controller) – realizează citirea și procesarea datelor din baza de date prin intermediul controller-ului care răspunde la comenzi utilizatorului prin partea de UI. Include toată logica de business necesară și realizează și randarea cu ultimele date a interfeței.
    • Partea de repository – layer principal de procesare între partea de controller și model și baza de date
    • Identity Framework – realizează partea de autentificare și securitate a contului în funcție de credențiale și rolul utilizatorului
    • Razor Views – se leagă cu MVC-ul pentru randarea paginilor din UI
Pe lângă cei trei piloni principali ai sistemului, mai trebuie menționat și faptul că acest proiect s-a desfășurat folosind și Docker care este un serviciu de rulare și mentenanță a containerelor pe care se rulează aplicații. În cadrul acestui proiect am folosit Docker pentru a rula RabbitMQ.
Implementarea soluției în discuție nu ar fi fost posibilă fără un server SQL manageriat de SSMS (SQL Server Management Studio). Acesta este indispensabil deoarece baza de date folosită este stocată aici.

5.2.1. HomeController și Identity Framework – Configurarea inițială
Acesta este primul pas necesar pentru a da drumul la cursul aplicației în discuție. Practic această etapă presupune adăugarea în baza de date a unui șofer, pregătirea unui traseu de către administrator și asignarea acestuia unuia dintre șoferi.
5.2.2. Clasa DataSeeder și serviciul Identity – Setup-ul inițial
Se pune la dispoziție, la prima folosire a aplicației, un cont predefinit manual pentru un „super" administrator cu niște credențiale stocate într-un tabel AspNetUsers. Parola pentru acest cont este securizată cu algoritmi de hasing de tip PBKDF2 care este pus la dispoziție de Identity Framework din cadrul ASP .NET Core 9. Acest cont poate fi folosit pentru o primă logare în ideea de a adăuga administratori "clasici" care, ulterior, pot popula baza de date cu alți administratori și șoferi. Contul predefinit este creat de clasa DataSeeder prin metoda SeedRolesAndAdminAsync() care inițializează rolurile și administratorul implicit în baza de date.
Partea de logare a unui administrator "clasic" presupune, în primul rând introducerea unor credențiale într-o pagină randată de tip Razor Page. După introducerea credențialelor și confirmarea logării prin apăsarea unui buton, se face un request HTTP către aplicația MVC prin serviciul SignIn pus la dispoziție de .NET Core 9. Parola este decriptată prin același algoritm de hashing PBKDF2 iar rezultatul este comparat cu credențialele stocate în același tabel adițional AspNetUsers. Partea de autentificare se face prin serviciile Identity Framework iar dacă validarea aceasta reușește, aplicația de MVC stimulează Razor Engine să randeze pe UI dashboard-ul specific unui administrator. În caz autentificare nereușită sau alte erori, tot Razor Engine randează pagina de login din nou cu afișarea mesajului specific de eroare pentru a pune la dispoziție utilizatorului să se logheze din nou.



5.2.3. HomeController.AddUser() și UserManager – Adăugarea unui șofer 
Aceasta se face foarte simplu prin elementele puse la dispoziție de Razor Pages și framework-ul Bootstrap de CSS. În acest fel, apăsarea butonului "Add", duce la apariția unei ferestre cu mai multe câmpuri unde se introduc datele șoferului. Acest tabel este implementat în JavaScript iar aceste date sunt preluate și procesate într-un .json și trimise mai departe la MVC printr-un request. MVC-ul le preia prin metoda HomeController.AddUser() și, prin intermediul unui engine de rutare, ele sunt trimise mai departe către partea de Model din MVC unde sunt parsate și procesate. Din zona de Model se fac niște request-uri necesare către baza de date din server-ul SQL pentru validarea datelor.
Crearea unui nou cont are loc doar dacă validările menționate anterior reușesc, moment în care se apelează serviciul de UserManager din framework-ul de Identity Core care plasează credențialele în AspNetUsers cu parola fiind supusă hashing-ului prin PBKDF2. După ce înrolarea aceasta este realizată, user-ului îi este atribuit rolul de șofer în tabela AspNetUserRoles, acest lucru fiind important în partea de validare la logări viitoare. În cele din urmă, framework-ul de Entity Core realizează înregistrarea noului utilizator în baza de date prin folosirea comenzilor specifice SQL.
Concretizarea operațiunii se face printr-un răspuns de tip HTTP către partea de JavaScript care a deschis formularul de înregistrare. Acest răspuns poate să fie cu cod de succes sau cod de eroare în funcție de rezultatul lanțului de operațiuni explicat anterior. Partea de UI poate să își facă un refresh iar cel care a vrut să introducă un alt user în baza de date va fi notificat de rezultat printr-o notificare oferită de librăria JavaScript SweetAlert2.

5.2.4. HomeController.AddCar() și Repository<Car> – Adăugarea unei mașini
Se realizează similar cu adăugarea unui șofer descrisă mai sus. Practic același dialog modal SweetAlert2 este folosit pentru a se adăuga datele mașinii (VIN, model, an fabricație etc.).  
Asigurarea unicității mașinii adăugate se realizează de fiecare dată când se completează tabelul de câmpuri. Practic, în funcție de VIN-ul mașinii, care reprezintă un identificator unic, această validare este posibilă. După ce se completează tabelul respectiv, se împachetează informațiile într-un .json și se trimite un request către MVC care, prin engine-ul de rutare, face forward către Controller-ul principal HomeController. Acesta preia request-ul prin metoda AddCar() și decodifică datele prin corelare cu partea de Model. Ulterior, se folosește Repository<Car> pentru abstractizare a comenzilor iar partea de Entity Framework core face parsarea tabelului Cars prin comenzi SQL pentru a verifica dacă noua adiție este deja existentă sau nu. Dacă mașina există deja, se trimite înapoi valoarea spre partea de client JavaScript, dacă nu se trimite un NULL.
Actualizarea bazei de date se face în câțiva pași bine stabiliți. După ce s-a primit răspunsul de mai sus, Controller-ul îl evaluează și în caz de nevoie de adăugare, se apelează comanda AddAsync() a Repository-ului care apelează partea de Entity Framework Core pentru a update baza de date cu o intrare nouă prin interfețe SQL. Acest lucru este confirmat și în nivelul de mai sus printr-un request pozitiv care aduce după el un mesaj de succes.

	
5.2.5. HomeController.AddRoute() și Entity Framework Core – Asignarea traseelor
În momentul în care un administrator apasă butonul de creare de rute noi se accesează interfața specifică prin Razor Pages. La fel ca și în situațiile anterioare, același tabel modal SweetAlert2 se afișează și câmpurile traseului pot fi completate. Trebuie ales un șofer căruia i se asignează traseul iar acest lucru se face prin afișarea unui meniu de drop-down de selecție din care se poate alege șoferul. Această listă drop-down se structurează prin parcurgerea unei instrucțiuni for care prin Entity Framework Core parsează toată table de AspNetUsers.
După ce se alege șoferul, mașina și se completează toate câmpurile, se confirmă crearea noului traseu care, la fel ca în cazurile anterioare, structurează un .json care este trimis la HomeController pentru procesare. Acest .json este decodificat și, tot prin Entity Framework Core, se verifică existența mașinii prin identificatorul unic VIN din tabela Cars și existența utilizatorului de driver prin identificatorul unic (CNP) din tabela AspNetUsers.
În caz afirmativ, se trimite un alt request de creare de traseu care este procesat de HomeController prin metoda AddRoute() și se creează un obiect de traseu de tip Route. Controller-ul adaugă un timestamp de creare în acest obiect și, mai departe, solicită acțiunile asociată butonului de adăugare din Repository prin metoda AddAsync(). Tot Entity Framework Core se folosesc comenzi de SQL_Insert pentru a adăuga noul traseul în tabela de Routes. De menționat că traseul este făcut unic prin corelarea prin chei străine cu o mașină și un șofer asignat. De asemenea traseul ajunge în starea de „Assigned" prima dată.



5.3.  Clasele AuthService, ObdService și RoutesPage – Inițializare și colectare de date
Aceasta etapa are legatura cu aplicatia mobile care ruleaza pe telefonul unui sofer si se referea la initalizarea unui traseu plus toate procesele implicate in colectarea datelor  de la masina prin OBD2 si de la GPS-ul telefonului.
5.3.1. AuthService și LoginPage – Inițializarea	
Partea de inițializare are legătură cu toate validările și procesele care se fac pentru a prelua, procesa și afișa în aplicația mobile a datelor, anterior, gestionate de aplicația Web și stocate în baza de date de pe serverul SQL.
Diagrama de inițializare poate fi vizualizată pe următoarea pagină


Entry Point-ul aplicației este în App.xaml.cs și este definit într-un obiect din clasa de bază Application. La lansarea aplicației se verifică dacă există un token de logare. În caz afirmativ, aplicația afișează pagina în curs a utilizatorului logat. În caz negativ, se afișează pagina principală de autentificare care este gestionată de clasa LoginPage.
Pagina de logare a aplicației în .NET MAUI este reprezentată prin niște fișiere xaml bazate pe obiecte moștenite din clasa ContentPage care reprezintă clasa mamă pentru afișarea oricărui template de pagină a framework-ului .NET MAUI. Pagina conține 2 entry-uri pentru user și parolă, un buton de logare care este atașat unui event cu handler-ul Clicked="OnLoginClicked" și un câmp de mesaje în caz de raportate de erori/probleme în această secvență de autentificare. Trebuie menționată metoda InitializeComponent() care, leagă entry-urile de username și parolă la codul din background punându-le la dispoziție ca variabile tipizate strict pentru a fi evaluate și utilizate ulterior.
Apăsarea butonului de Login apelează metoda OnLoginClicked care duce la verificările entry-urilor de user și parolă și validarea sau invalidarea acestora printr-o verificare inițială dacă câmpurile sunt sau nu goale. Un rol extrem de important în această fază îl are constructorul AuthService care are rolul de a trimite un request HTTP în caz de validare a autentificării, proces realizat de obiectul HttpClient care are hardcodată adresa de endpoint a aplicației Web. 
Request-ul merge prin internet și ajunge la serverul Kestrel care manageriază aplicația Web prin portul de localhost 7292. Acest request vine la pachet cu validarea user-ului și a parolei care sunt încapsulate într-un obiect LoginModel care este codificat și încapsulat într-un .json care este atașat request-ului principal către aplicația Web. 
Validarea credențialelor
Pașii de validare ai credențialelor se fac în aplicația MVC după ce request-ul de logare a fost primit și procesat. Credențialele introduse ajung într-un obiect LoginDto care este procesat în partea de Model al MVC. Username-ul și Parola sunt extrase și validate prin serviciul de UserManager al Identity Framework.
Toți acești pași asigură că prin logarea unui utilizator valid în aplicația mobile, se accesează aceeași bază de date din serverul SQL care este legată la MVC-ul aplicației Web. Metoda FindByEmailAsync parsează tabela AsptNetUsers și dacă un entry cu email-ul respectiv este găsit, se trece la validarea parolei. În acest sens, se extrage valoare hașurată a parolei PasswordHash stocată în baza de date deoarece parola fără hashing nu este stocată nicăieri pentru a evita probleme de securitate. Pentru valoarea PasswordHash se aplică algoritmul PBKDF2 de dehașurare iar apoi se compară rezultatul cu parola stocată în obiectul LoginDto. După ce parola este validată, metoda GetRolesAsync parsează tabelele legate AspNetUsers și AspNetRoles pentru a stabili rolul celui care s-a logat în aplicația mobile.
Crearea unui token JWT (JSON Web Token) reprezintă pasul care urmează după stabilirea rolului utilizatorului proaspăt logat. Este important de știut că se stabilesc anumite claim-uri despre cel logat prin creeare unor obiecte precum JwtRegisteredClaimNames.Sub care conține identificatorul unic al celui autentificat sau ClaimTypes.Role care încapsulează rolul celui logat. Token-ul JWT este securizat prin algoritmi de criptare HMAC-SHA256 și conține date precum: server-ul care a eliberat token-ul, audiența pentru care acesta a fost eliberat, date despre utilizator salvate în obiecte de tip Claim, perioada de expirare și credențiale pentru parte de criptare. 
După ce token-ul este creat, Controller-ul din MVC trimite un răspuns HTTP de notificare a părții de .NET MAUI. Metoda AuthService.LoginAsync primește răspunsul și decodifică .json-ul și salvează token-ul sub forma unui obiect de tip TokenResponse care este stocat în telefon prin SecureStorage din .NET MAUI Essentials.


Afișarea traseelor este posibilă după ce tokenul este securizat și stocat în memoria telefonului. Practic handler-ul LoginPage, apelează metoda Navigation.PushAsync(_routesPage) a clasei Navigation care afișează pagina RoutesPage care moștenește ContentPage și care afișează traseele disponibile pentru șoferul logat.
Important de menționat este că, înainte ca traseele cu detaliile respective să fie randate în interfața UI din .NET MAUI, trebuie făcut un HTTP Get request către aplicația Web. Acest request este făcut tot de obiectul HttpClient care trebuie configurat cu token-ul presalvat pentru autentificare. Comunicarea prin HTTP din acest scenariu este asigurată de metoda RouteService.GetDriverRoutesAsync() iar reactualizarea paginii de trasee cu ultimele detalii venite de la MVC-ul din cealaltă parte se face prin metoda LoadRoutesAsync().
Pe partea de MVC, după ce request-ul Get este primit, acesta este procesat prin extragerea token-ului din headerul Authorization și evaluarea parametrilor acestuia (timestamps, claims, data expirare etc.). Dacă totul este în regulă, se trece la nivelul de Repository unde se folosește metoda GetAll() pentru a face un query la baza de date pentru returnare unui traseu cu toate detaliile aferente: id traseu, nume, ora începere, locație pornire, locație terminare, seria mașinii, data încheiere. Aceste date sunt structurate într-un .JSON și trimite înapoi spre aplicația mobile prin alt request HTTP.
Înapoi în partea de Mobile, request-ul HTTP este primit iar .JSON-ul este deserializat iar datele traseului sunt stocate în obiectul List<RouteDto>. Pentru fiecare obiect de acest tip, metoda LoadRoutesAsync initializează un obiect Points de tipul ObservableCollection<GpsLogEntry> care reprezintă un log pentru coordonatele GPS care vor fi preluate de la telefon la începerea traseului.






5.3.2. ObdService și RoutesPage – Colectarea datelor
Această etapă este nucleul întregii aplicații de flote pentru mașini. Fără această etapă, nu ar exista niciun sens ca aplicația să existe. Fără această etapă nu ar fi posibilă nici diagnoza pe mașini, nici orice posibilitate de tracking a șoferilor și urmărirea traseelor. 



Prin apăsarea butonului de „Start" se apelează handler-ul de buton OnStartClicked care leagă un obiect sender care conține butonul care a fost apăsat și se leagă de CommandParameter care conține id-ul traseului care vrea să se înceapă.
Partea de permisiuni ale device-ului care se leagă la ELM327 prin Bluetooth (în cazul nostru, telefonul șoferului) este gestionată de metoda EnsureBluetoothAndLocationPermissions care, făcând parte din framework-ul nou .NET MAUI, are capacitatea de a prezenta user-ului un pop-up pe care acesta îl poate folosi pentru a acorda permisiunile necesare telefonului de a se conecta prin Bluetooth la ELM327. Tot metoda EnsureBluetoothAndLocationPermissions folosește în mod asincron metoda LocationWhenInUse care rulează constant în background și este responsabilă de acordarea permisiunilor de colectare de date GPS ale telefonului. Practic EnsureBluetoothAndLocationPermissions încapsulează partea de asigurarea de orice tip de permisiune e necesară între aplicația mobile, telefonul mașinii și OBD2 pregătind terenul pentru pasul următor.
Request-ul HTTP către aplicația Web este inițiat de metoda UpdateRouteStatusAsync care funcționează tot asincron, și care încapsulează detaliile traseului care se dorește să se înceapă. Acest request este trimis și este preluat de Controller-ul din MVC care face validările necesare pentru token-ul JWT din AuthHandler (lucru detaliat în subcapitolul anterior). Odată ce token-ul este validat, se apelează partea de Repository prin metoda GetAsync și se structurează comenzile SQL necesare pentru a accesa baza de date din serverul SQL și a o parsa pentru a ajunge la traseul căutat. Trebuie menționată metoda UpdateStatus care face verificările și acțiunile necesare pentru schimbarea statusului din din „Assigned" în „Started". O validare foarte importantă este realizată de route.UserId != User.FindFirstValue(ClaimTypes.NameIdentifier) care asigură că un traseu poate să fie modificat doar dacă este început de un șofer. Odată ce statusul a fost update în baza de date, metoda LoadRoutesAsync() ajută la reîmprospătarea interfeței în aplicația mobile iar toate datele noi apar pe interfață. 
Începerea unui traseu presupune reprezentarea la nivel de cod a traseului ales care se face prin variabila _currentRoute care este o referință a traseului început și se va folosi pe parcurs pentru asocierea dintre traseul început și datele GPS și cele de la ELM327. 
Cele două puncte principale în colectarea datelor sunt următoarele:
    • Colectarea datelor de diagnoză de la OBD2 
    • Conexiunea device-ului prin Bluetooth realizată prin metoda _obdService.Connect()
    • Colectarea datelor de la ELM327 care începe prin apelarea _obdService.StartMonitoring()
    • Partea de logare de date GPS activată prin StartGpsLogging()

5.4. Clasa RabbitMqService – Publicarea mesajelor
În această fază a întregului circuit, se realizează publicarea de mesaje în broker-ul de mesaje intermediar RabbitMQ. Odată ce șofer-ul a pornit traseul, începe colectarea de date constantă pe cele două canale menționate mai sus (date de diagnoză de la ELM327 și date GPS de la telefon). În mod teoretic, un broker intermediar de mesaje nu ar fi necesar deoarece, așa cum există comunicare între aplicația mobile și web în secvența de logare și alegere traseu descrise în fazele precedente, la fel se pot transmite date către aplicația MVC și prin alte metode. Un broker de mesaje intermediar ajută pentru optimizarea performanței și reducerea riscului de malfuncționare deoarece, separându-se partea de publicare de mesaje (RabbitMQ) de partea de consum de mesaje (Worker) nu are întregul sistem de suferit în cazul în care una din componente funcționează greșit. 
Arhitectura RabbitMQ 
Trebuie menționat în primul rând că tot framework-ul de Rabbit este compatibil cu ASP .NET Core 9 și poate integra ușor cu IDE-ul de Visual Studio prin instalarea pachetului RabbitMQ.Client prin serviciul de NuGet Package Manager.
La baza arhitecturii broker-ului de mesaje RabbitMQ, stă clasa RabbitMqService definită ca un singleton în fișierul principal MauiProgram.cs din orice aplicație generată cu framework-ul .NET MAUI. Fiind un singleton, doar o singură instanță a clasei respective poate să apară în tot proiectul. Clasa conține toate interfețele necesare și toată logica necesară gestionării mesajelor care se stochează în cozile RabbitMQ. Protocolul de comunicare folosit de RabbitMQ este AMQP (Advanced Message Queuing Protocol) care asigură mecanismele necesare pentru publicarea și consumul mesajelor și întreținerea cozilor respective.
Elemente principale din clasa RabbitMqService, ar fi câteva dar menționăm în, primul rând, o instanță de ConnectionFactory care conține parametri și metode necesare pentru conexiunea la cozile din RabbitMQ. La fel de important este un obiect Iconnection care reprezintă un socket care reține detaliile conexiunii TCP la serverul de RabbitMQ. Pe lângă asta, elementul de Iconnection ajută la reglarea transmiterii frame-urile din cadrul protocolului AMQP. Interfațarea dintre un mesaj și o conexiune se realizează prin folosirea unui canal de tip Ichannel care încapsulează partea de publicare sau consum de mesaje inspre și dinspre cozile de RabbitMQ. Relevant pentru o instanță unică de RabbitMQ este și parametrul Queue intern în care se țin valorile obținute de la GPS-ul telefonului. A nu se confunda cu queue-ul principal RabbitMQ unde sunt stocate mesajele abstractizate pe arhitectura RabbitMqService.
De menționat este că, în aplicația de față, o conexiune spre RabbitMQ se face cu ip-ul hardcodat 192.168.255.187 deoarece acesta este același ip folosit și de aplicația web și de Worker. Acestea fiind spuse, pentru sistemul să poată publica datele pe care le colectează de la mașină, respectiv GPS-ul telefonului, trebuie să ne asigurăm că o conexiune a telefonului șoferului la această adresă este posibilă. Trebuie menționate și porturile 15672 care este adresa default pentru accesarea UI-ului RabbitMQ și portul 5672 necesar protocolului AMPQ pentru a rula.
Pașii de funcționare RabbitMQ
Prima dată este nevoie de logarea în RabbitMQ a tuturor aplicațiilor care folosesc acest serviciu. În cazul aplicației noastre este aplicația .NET MAUI pentru publicare de mesaje și Worker-ul pentru consum de mesaje. Credențialele predefinite există definite în codul de implementare dar trebuie ca baza de date RabbitMQ să conțină un utilizator predefinit cu aceste credențiale. Dacă acest lucru există, accesul la serviciu este posibil. 
Trebuie menționat că o instantierea unui obiect RabbitMqService nu duce, de la sine, la instantierea unei noi conexiuni, automat. O conexiune se instantiază doar când se detectează prima tentativă de a publica un mesaj în coada de RabbitMQ. Aceasta se numește inițializare „leneșă" și vine la pachet cu avantajul că timpul de start al aplicației este redus și că nu există consum nenecesar în cazul lipsei comunicării. După crearea conexiunii, se creează un canal care este validat și legat la conexiunile socket-ului TCP prin configurarea pentru a adopta frame-urile protocolului AMQP. Canalul primește un id unic iar crearea lui este simplă și în timp redus. Acești pași sunt implementați în metoda EnsureChannelAsync() care asigură existența unei conexiuni și a unui canal valid înainte de publicarea mesajelor.
Crearea cozii de mesaje este un pas primordial în structura de RabbitMQ. Aceasta se creează cu metoda QueueDeclareAsync care ia legătura cu broker-ul prin declararea unui frame AMQP care se face prin Queue.Declare. Ulterior se așteaptă răspuns de la broker dacă e posibilă crearea unei cozi. În caz afirmativ, se fac inițializările de memorie și alocările necesare iar coada este creată și instantiată.   
Ultima etapă reprezintă, efectiv, publicarea de mesaje care presupune popularea cozii de mesaje din RabbitMQ cu mesajele care conțin datele de la mașină și la de la GPS-ul telefonului. Acest lucru nu ar fi posibil fără metoda PublishAsync() care are doi parametri care merită menționați: un payload care este de tipul Object pentru a putea înmagazina orice tip de obiect structură care poate fi depozitat în coada de Rabbit. În cazul nostru ajută la faptul că trebuie procesate date de diferite tipuri și dimensiuni când vine vorba de OBD2 și GPS. Al doilea parametru al acestei metode este un string routingKey care este cheia sau numele cozii pe care sistemul o folosește.
După ce coada se creează, ea trebuie populată cu mesaje. Acest lucru nu se face direct ci în niște pași simpli:
    • Mesajele de tip Object se decodifică și se structurează în niște string-uri JSON printr-o serializare permisă de JsonSerializer
    • String-ul rezultate se preia și se codifică în format UTF-8 într-un array de bytes deoarece acesta este formatul cerut de mesajele din RabbitMQ
    • BasicPublishAsync realizează popularea cozii cu mesajul rezultat în pasul anterior
Metoda BasicPublishAsync realizează publicarea mesajului în coada RabbitMQ în mod asincron însă ea este apelată în mod „await" de către Metoda principală. Acest lucru înseamnă că metoda este apelată și așteaptă confirmarea publicării mesajului dar nu așteaptă confirmarea broker-ului cum că mesajul s-a primit. Această metodă de lucru dinamică ajută în cazul proiectului de față deoarece o flotă cu multe mașini care colectează date simultan presupune un trafic foarte mare pe serverul de Rabbit iar absența verificării fiecărui mesaj în parte de către broker aduce cu sine o viteză crescută de lucru. Dezavantajul în această situație este faptul că unele mesaje se pot pierde fără ca publisher-ul (aplicația .NET MAUI în cazul nostru) să fie notificat. Cu toate acestea, traficul intens din proiectul de față reduce impactul unor mesaje pierdute deoarece datele sunt reactualizate frecvent fără a crea rate-uri vizibile în aplicație.
Conexiunea către RabbitMQ, odată creată la lansarea server-ului rămâne în uz cât timp aplicația funcționează și conexiunea la internet se menține. Acest lucru vine cu avantaje datorită faptului că, oricând, este cazul să se reia publicarea de mesaje, se formează doar un frame nou cu protocolul AMPQ și se publică în coada deja configurată și disponibilă. Dezavantaje pot apărea în caz de pauze în conexiunea de internet sau trecerea de la un network la celălalt. În aceste situații pot apărea probleme de malfuncționare temporare care se rezolvă, în schimb, după ce inițializările principale au loc. Implementarea actuală prioritizează performanța și permite posibile rate-uri în trimiterea unor mesaje în situații excepționale. Aceste neajunsuri sunt acceptate, însă, deoarece, după cum spuneam, comunicația este de așa natură și rata de apariție e așa redusă încât acestea pot trece, practic, neobservate. Metoda DisposeAsync() este responsabilă de eliberarea resurselor conexiunii și canalului la oprirea aplicației.

5.5. Clasa Worker și AsyncEventingBasicConsumer – Consumul mesajelor
A patra etapă în lanțul funcționării acestei aplicații este reprezentată de partea de consum sau extragere a mesajelor din serverul RabbitMQ și introducerea sau, folosind termenul tehnic specific, injectarea acestora în baza de date din serverul SQL. 
Acest proces este realizat de către o aplicație implementată tot cu ajutorul .NET Core 9 care se cheamă Worker și care lucrează în background independent de activitatea aplicației web sau a broker-ului RabbitMQ. Clasa Worker extinde BackgroundService, iar acest lucru vine la pachet cu niște avantaje care merită sumarizate: 
    • Aplicația principală de web funcționează mai rapid și mai eficient
    • în eventualitatea unui crash al worker-ului, aplicația web continuă să funcționeze exceptând partea de actualizare de mesaje
    • mai multe instanțe ale unui worker pot fi dezvoltate și rulate pe servere diferite RabbitMQ 
 Worker-ul extinde clasa BackgroundService care ajută la susținerea proceselor pe termen lung. Pe lângă aceasta, worker-ul are fișier separat de configurare .json, mecanisme de recovery și algoritm de graceful shutdown.
Nu vom intra în detalii amănunțite în ceea ce privește Worker-ul la nivel de implementare însă merită menționate niște componente și interfețe principale. În cazul proiectului de față, Worker-ul în discuție folosește un obiect de tip ILogger<Worker> instanțiat de niște mecanisme de logging incluse în ASP .NET Core 9. În continuare, o interfață IServiceScopeFactory este folosită pentru a defini o zonă în care datele citite de pe RabbitMQ se injectează, în cazul aplicației de față, acesta fiind baza de date de pe server-ul SQL. Pe urmă, exact ca în cazul publisher-ului din aplicația mobile se definește o conexiune și un canal cu aceleași valori în ambele părți. Nu în ultimul rând, partea de interacțiune cu RabbitMQ este determinată de un obiect de tipul AsyncEventingBasicConsumer care funcționează în mod event-driven în sensul că se bazează pe niște callback-uri care sunt lansate atunci când RabbitMQ transmite mesaje. Acest lucru scutește Worker-ul să facă polling pe serviciul de RabbitMQ pentru a afla statusul mesajelor, acest lucru crescând performanța de lucru.
Inițializarea serviciului de Worker are loc prin task-ul de StartAsync(), etapă care se execută rapid înainte de apelarea funcției principale main() a întregului sistem. În această etapă se stabilește conexiunea cu RabbitMQ împreună cu inițializarea canalului și a altor parametri importanți și determină Worker-ul să asculte coada de RabbitMQ. Această etapă trebuie executată rapid și o singură dată. 
Ideea de bază din spatele modularității Worker-ului este faptul că funcționarea acestuia trebuie dirijată în funcție de starea serviciului de RabbitMQ. În primul rând, inițializarea unei conexiuni ConnectionFactor se face definind parametrul HostName cu valoarea "localhost" la fel ca și conexiunea creată, anterior, în aplicația de .NET MAUI. Tot aspectul de funcționare asincronă este demonstrat și de metodele CreateConnectionAsync(), respectiv CreateChannelAsync() care folosesc keyword-ul await pentru a amâna execuția acestora până pasul precedent se finalizează. CreateConnectionAsync() realizează și inițializează și un handshake al protocolului AMPQ cu partea de RabbitMQ însă acest lucru se întâmplă doar după ce obiectul ConnectionFactory a fost creat și inițializat cu numele "localhost". Pe urmă, metoda CreateChannelAsync() are rolul de a creea un canal legat la coada de RabbitMQ.
Trecând la un pas important în algoritmul de implementare, ajungem la crearea și inițializarea cozii RabbitMQ. În proiectul de față, coada se creează și inițializează prin metoda QueueDeclareAsync() care primește niște argumente specifice. Important de notat în legătură cu coada de pe RabbitMQ este faptul că aceasta este prelucrată sub principiul idempotentei. În situația de față, acest principiu stă la baza colaborării dintre Worker și aplicația de mobile care accesează coada de RabbitMQ. Logica de funcționare cuprinde următoarele situații:
    • Dacă coada este deja declarată cu aceiași parametri de către aplicația mobile, Worker-ul redeclară coada fără nicio problemă
    • Dacă coada este deja declarată cu alți parametri de către aplicația mobile, se aruncă o excepție de funcționare și inițializarea nu are loc
    • Dacă coada nu există, ea este creată de Worker și inițializată corect
De notat ar fi că prin principiul idempotentei deja menționat, punctele de mai sus se aplică și invers, însemnând că și aplicația mobile respectă aceleași reguli dacă Worker-ul a fost primul care a inițializat coada. Ce s-a descris adineauri reprezintă un imens avantaj pe partea de eficiență de funcționare cât și pe partea de robustețe a SW-ului creat deoarece reduce fricțiunea și dependințele de funcționare între Worker și aplicația de mobile.



Configurarea Worker-ului este următorul pas în acest proces. Detalii care merită menționate ar fi faptul că se folosește metoda AsyncEventingBasicConsumer() care ajută la crearea unui handler event-driven care are rolul și posibilitatea de a efectua operațiuni intrare-ieșire pe coada de mesaje fără a bloca thread-ul care face dispatch la mesajele de pe coadă. La detectarea unui mesaj de către acest handler, se apelează ReceivedAsync cu doi parametri dintre care foarte important este parametrul ea care reprezintă o structură de tipul BasicDeliverEventArgs are încapsulează metadata-ul și conținutul mesajului. După ce acest mesaj se primește, consumer-ul folosește metoda ToArray() pentru a converti informațiile din mesaj într-un array de bytes. În cele din urmă, acest array de bytes este decodificat cu format-ul UTF-8 într-un string, acesta reprezentând exact mesajul inițial al aplicației mobile încapsulat intru-un .JSON. Urmează, apoi, partea de deserializare .JSON prin metoda System.Text.Json.JsonSerializer. După deserializare, datele se introduc într-un obiect DTO (de tipul CarData sau RouteGpsLogEntry) deja creat pentru aceeași structură ca și mesajul venit în ceea ce privește parametrii și tipul acestuia. Dacă DTO-ul n este creat corespunzător, se aruncă o excepție și transferul respectiv se anulează.


Interacțiunea Worker-ului cu baza de date este un alt aspect important care trebuie clarificat. Există o problemă în acest aspect deoarece baza de date este un obiect mare de tipul DbContext (concret WebAppDbContext), după cum s-a menționat și anterior, care asigură operațiunile de tracking ale schimbărilor, efectuarea schimbărilor în baza de date, conexiunile bazei de date și, nu în ultimul rând, se asigură că, la orice procesare, este luat în calcul doar o singură instanță a unui obiect procesat. Toate aceste mecanisme de funcționare ridică o problemă reală în interacțiunea dintre o bază de date realizată cu ajutorul Entity Framework Core și un Worker care reprezintă un singleton fără posibilitatea de a se instantia de mai multe ori.
Pentru a rezolva problema descrisă mai sus, în primul rând, se folosește metoda CreateScope() care are rolul de a crea un container nou de tipul dependency injection care moștenește serviciile instanței unice de Worker și, în același timp, face o copie a serviciilor necesare din obiectul DbContext. Mai departe, se face o cerere către container-ul provizoriu creat pentru acces-ul la repository-ul propriu (IRepository<T>) care abstractizează interacțiunea cu baza de date la fel ca și în aplicația principală de MVC. Am menționat anterior că noul container creat este provizoriu și așa și este. Asta pentru că, după ce operațiunile necesare sunt efectuate pentru respectivul scope al respectivului mesaj, container-ul este șters cu metoda Dispose. În acest fel se elimină pericolul pierderii de informații sau a problemelor de performanță cauzate de flow-ul constant de date de la mașină.
Partea de mapare cu baza de date presupune popularea entităților bazei de date cu datele din .JSON-ul deserializat. Acest lucru se face printr-un algoritm de parsare a mesajului .JSON. Pentru unele date este necesară o conversie a tipului de date. De exemplu, latitudinea și longitudinea pentru coordonatele GPS ajung de tipul double în partea de .JSON dar se folosește metoda ToString() pentru a le converti și stoca ca și string-uri în entitatea CarData. Pentru popularea unui traseu, trebuie populată tabela de trasee cu datele din .JSON și acest lucru se face prin cheia străină RouteId care trebuie să existe în tabelul Routes. În caz contrar, framework-ul aruncă o excepție.
Principiul de populare a unei entități din baza de date este descris în continuare. Trebuie reamintit container-ul provizoriu creat prin CreateScope() descris anterior. Respectivul container implementează interfața IRepository<T> unde T este clasa de obiecte care populează baza de date. Această interfață este împărtășită de aplicația web și de Worker și, prin metoda AddAsync(entity) se adaugă entitatea „entity" în tabela corespunzătoare care este un obiect de tip DbSet<T> unde T reprezintă clasa respectivă. De exemplu, adăugarea unei entități în tabela CarData se face prin popularea unui obiect de tip DbSet<T> unde T este CarData. Această schimbare se reflectă într-un ChangeTracker și prima stare după procesul anterior va fi „Added". Imediat după AddAsync, urmează SaveChangesAsync() care scaneză acest ChangeTracker și detectează noul obiect T, moment în care se generează instrucțiunile SQL necesare pentru introducerea entității T în tabelul corespunzător. Acest pachet de instrucțiuni ajunge la serverul SQL prin protocolul TDS și, astfel, se execută parsarea entității de introdus, se validează existența cheii străine care reprezintă rândul unde trebuie introdusă noua entitate, se introduc datele pe rândul respectiv și se returnează statusul operațiunii.



	Interacțiunea dintre Worker și brokerul RabbitMQ în acest proiect, se face în câțiva pași distincți. În primul rând am stabilit să folosim metoda de acknowledgement automat al mesajelor venite la de la broker. Acest lucru înseamnă că orice mesaj sosește din partea RabbitMQ este, automat, recunoscut de către Worker și nu este nevoie ca acesta să trimită vreo confirmare înapoi la RabbitMQ. Având în vedere, că această aplicație presupune flux constant de date, am ales această metodă deși, în lipsa performanțelor necesare, există posibilitatea ca Worker-ul să nu facă față și unele mesaje să se piardă. Cu toate acestea, fluxul mare de date, este reconfortant în sensul că, și dacă un frame se pierde, vine altul imediat după iar informația este reîmprospătată oricum. 
După ce se stabilește tipul comunicării explicat mai sus, trebuie ca Worker-ul să se „înregistreze" la singura coadă disponibilă în RabbitMQ, în cazul nostru. Această etapă presupune trimiterea un frame inițial AMPQ către broker care spune, practic, că Worker-ul vrea să se „lege" la coada respectivă. După ce RabbitMQ primește acest request, îl procesează și îi trimite un răspuns de confirmare Worker-ului. Din acest punct înainte, de fiecare dată când RabbitMQ primește un mesaj în coadă, îl trimite mai departe către Worker sub forma unui frame AMPQ fără să aștepte vreo confirmare de la acesta, deoarece am optat pentru acknowledgement automat după cum am explicat mai sus.
În final, trebuie să descriem partea de shutdown în cazul Worker-ului. Algoritmul implementat în câțiva pași se numește Graceful Shutdown și se face în felul următor: comanda de shutdown setează un obiect de tip cancellationToken din cadrul metodei ExecuteAsync() care se ocupă de consumul de mesaje implementat într-un while loop. Când acest token este setat, loop-ul se întrerupe și metoda se încheie. Framework-ul ASP .NET are grijă, ca ultima execuție în materie de consum de mesaje să se finalizeze înainte să facă shutdown la tot Worker-ul.

5.6. HomeController și Razor Views – Vizualizarea în aplicația Web
În acest subcapitol vom descrie implementarea aplicației care se ocupă de cum arată interfața în aplicația web atât pentru administratori cât și pentru șoferi. Pentru fiecare în parte, vom descrie pașii prin care se trece pentru a extrage datele din baza de date, a le prelucra și, în cele din urmă, a le afișa pe interfața corespunzător.

5.6.1. Identity Framework și atributul Authorize – Autentificarea Role-Based
După cum am descris și la începutul subcapitolului 5.2.1, prima fază în procesul de vizualizare pe web a datelor, este autentificarea. Am menționat că această aplicație folosește Identity Framework pus la dispoziție de .NET Core 9. Identity Framework se ocupă de partea de verificare a rolului celui care accesează aplicația și de autorizarea accesului sau nu.
Atunci când este apăsat butonul de „Login", Controller-ul din MVC declanșează task-ul Index care are atributul „Authorize" pus la dispoziție de Identity Framework. Acest atribut îi dă posibilitatea task-ului de Index să autorizeze un user pentru acces dacă autentificarea a reușit. În contextul de față, verificarea cont-ului se face prin metoda FromResult(User) care setează flag-ul IsAuthenticated în cazul autentificării cu succes. După ce autentificarea reușește, folosim IsInRole() pentru a redirecționa utilizatorul către pagina de administrator, respectiv șofer, în funcție de situație. 

5.6.2. Admin.cshtml și admin.js – Fereastra de administrator
Pentru a afișa fereastra de administrator, aplicația execută câțiva pași bine definiți. 
Partea de extragere a informațiilor din baza de date se face în cadrul task-ului principal Admin() din HomeController care este autorizat prin atributul Authorize care stabilește ca task-ul să fie executat când un user cu rol de admin se loghează. În cadrul task-ului Admin(), au loc niște apeluri către baza de date prin metoda refolosită GetAll(). Această metodă apelează Repository-ul din cadrul MVC care, la rândul lui abstractizează această comandă și apelează apoi Set<T>().ToListAsync(). Prin acești pași, se interacționează cu Entity Framework Core care translatează asta în comenzi SQL care extrag informațiile din baza de date și le plasează într-o listă în mod asincron.
Partea de afișare a informațiilor pe dashboard-ul administratorului se face prin colectarea tuturor informațiilor din baza de date într-un obiect de tip DTO (Data Transfer Object), concret DisplayDTO. Trebuie reamintit că pe interfața administratorului se afișează mașina și datele de la Obd2 legată de un user și un traseu. De asemenea trebuie afișate, în concordanță, și datele GPS de la telefonul mașinii pentru a reconstrui traseul alături de niște timestamp-uri. La nivel de implementare, ca și flow, trebuie menționați următorii pași atunci când administratorul deschide un traseu pentru a-l vizualiza:
    • Se accesează tabela CarData și se caută traseul respectiv după RouteId
    • Prin acest RouteId se ajunge la id-ul de user și id-ul de mașină
    • Se accesează, apoi, tabelele de user și mașină și se extrag informațiile necesare pentru a fi afișate
Înainte de vizualizare, datele încapsulate într-un DTO, cum este descris mai sus, se introduc într-un obiect de tip ViewBag care va fi, la rândul lui, input-ul pentru partea de Razor Pages randate prin Admin.cshtml.


Adăugarea unui user este foarte importantă deoarece are legătură cu toate pop-urile care apar atunci când un administrator vrea să adauge un user apăsând butonul de "Add User". Trebuie scoasă în evidență librăria SweetAlert2 folosită în aplicația noastră pentru asta. Practic, această librărie pune la dispoziție interfețele necesare în JavaScript pentru a face următoarele:
    • Deschiderea unui pop-up nou cu câmpurile de nume pentru user când administratorul apasă pe "Add User"
    • Colectarea datelor într-un obiect pe care administratorul le introduce legat de numele șoferului
    • Folosim librăria Jquery pentru a încapsula datele din obiectul Java script într-un .JSON care este trimis la end point-ul /addUser din backend pentru procesare
    • Când informația ajunge în backend, se apelează HomeController.AddUser() care creează un record de Driver 
    • În cele din urmă, se trimite un mesaj de confirmare pentru frontend iar pagina își face refresh pentru a afișa noul șofer
Adăugarea unei mașini se face, practic identic cu pașii descriși mai sus.
Adăugarea unui traseu este de o complexitate puțin mai ridicată și merită descrisă separat. În primul rând, când este deschis modalul de adăugare de traseu prin apăsarea "Add Route", trebuie colectați șoferii și mașinile disponibile pentru a le afișa în lista de dropdown. Informațiile se primesc de la backend într-un .JSON iar, în momentul care este făcută o selecție, se face o mapare după entitate (user,mașină). Apoi când toate datele sunt introduse iar traseul este confirmat, se realizează pașii de mai sus pentru popularea tabelei Routes. Diferența este că prin mapările menționate anterior, se pot completa și câmpurile id-ului mașinii și șoferului care reprezintă chei străine dintre tabela principală de Routes și tabelele secundare de Users și Cars. 

5.6.3. LoginPage și RoutesPage – Fereastra de șofer
Acest subcapitol acoperă partea de vizualizare a datelor și folosirea aplicației mobile de către șoferi. Vom trece prin partea de logare, afișare a traseelor, începerea și terminarea unui traseu.
Logarea în aplicație se face la fel ca și pentru administratori în cazul aplicației web și anume: se introduc credențialele de username și parolă și se apasă butonul „Login". În momentul în care acest buton este apăsat, se declanșează un event de OnLoginClicked care apelează AuthService.Login() care trimite un request către obiectul /api/auth/login din backend-ul aplicației. 
Interacțiunea dintre aplicația de mobile și backend-ul aplicației web este foarte asemănătoare cu interacțiunea dintre frontend și backend. Orice request de logare se trimite printr-un request HTTP care este serializat sub forma unui .json însoțit de un token de autentificare JWT.



	Afișarea datelor GPS ale telefonului se face prin metoda LogPointAsync() care ia datele și le stochează în variabile care sunt, ulterior, introduse în log-ul pentru datele GPS ObservableCollection<GpsLogEntry>. Calea este directă și eficientă iar stocarea se face real-time pe măsură ce traseul se parcurge. 
Randarea traseelor pe ecran se face prin metoda LoadRoutesAsync() care preia datele din backend și le încarcă pe ecran. Prima dată se face un clear la tot ce era pe ecran înainte. Apoi, se apelează RouteService.GetDriverRoutesAsync() care face un request la backend și, mai departe, se populează o listă cu traseele curente. Afișarea se face printr-un model de binding al maui care este rapid și permite un refresh rate foarte bun. Important de menționat este că toate datele, înainte de a se afișa pe ecranul telefonului, sunt stocate într-o colecție de obiecte denumită ObservableCollection. Acest Observable Collection conține toate detaliile despre traseu: punct plecare, pornire, timestamp, date GPS etc.
